# ADR-003: Shared Reverse Proxy for Standard Port Access

**Date:** 2026-03-27
**Status:** Proposed
**Decision Makers:** Project maintainers
**Technical Story:** After ADR-001 assigned unique high-numbered ports to each project, users must access sites via `https://local.tvmatchen.nu:10091` instead of `https://local.tvmatchen.nu`. A shared reverse proxy on ports 80/443 should route by `Host` header to the correct project's nginx container.

---

## Context

ADR-001 solved simultaneous multi-project operation by assigning each project a unique port block (e.g., HTTP=10090, HTTPS=10091 for tvmatchen). This works, but it forces developers to remember port numbers and type URLs like `https://local.tvmatchen.nu:10091`. It also breaks WordPress sites that store their `siteurl` and `home` options as `https://local.tvmatchen.nu` (no port) -- any redirect or absolute URL generated by WordPress will point to port 443, which nothing is listening on.

The goal is a single shared process that binds ports 80 and 443 on the host and routes incoming requests to the correct project based on the `Host` header.

### Current State (What Exists Today)

```
Browser → https://local.tvmatchen.nu:10091 → host port 10091 → tvmatchen_nginx:443 → tvmatchen_wordpress:9000
Browser → https://porto.meetup.local:10061 → host port 10061 → meetup_nginx:443 → meetup_wordpress:9000
```

Each project has:
- Its own Docker Compose stack with an isolated `{name}_wordpress_network` bridge network.
- An nginx container (`{name}_nginx`) that handles SSL termination and PHP-FPM proxying.
- SSL certs at `wordpress-projects/{name}/ssl/cert.pem` and `ssl/key.pem`.
- A domain stored in `config.json` (e.g., `local.tvmatchen.nu`, `porto.meetup.local`, `local.casinobonukset.com`).
- Unique host-mapped ports via `.env` (HTTP_PORT, HTTPS_PORT).

### Constraints

- **Local dev tool**: simplicity and reliability matter more than production-grade features.
- **Team familiarity**: the team knows nginx well; Traefik/Caddy would be new.
- **Automatic configuration**: starting/stopping a project must update the proxy without manual steps.
- **No interference**: existing per-project nginx configs (PHP-FPM, rewrites, subfolder routing) must not change.
- **Mixed domain patterns**: must handle `local.{name}.test`, `local.{name}.nu`, `{name}.meetup.local`, `local.{name}.com` -- no single wildcard covers all.
- **macOS primary target**: Docker Desktop for Mac.
- **Ports 80/443 on macOS**: Docker Desktop can bind these without sudo since it runs in a VM.

---

## Options Considered

### Option A: Do Nothing

Continue using high-numbered ports. Users bookmark `https://local.tvmatchen.nu:10091`.

**Pros:**
- Zero implementation effort.
- No new moving parts.

**Cons:**
- WordPress `siteurl`/`home` mismatch causes redirect loops and broken assets when the DB stores port-less URLs.
- Poor developer experience -- remembering ports per project is error-prone.
- Copy-pasting URLs to colleagues or documentation includes non-standard ports.

**Complexity:** None

### Option B: Standalone Nginx Reverse Proxy with Generated Config

Run a single nginx container (its own `docker-compose.yml` at the repo root) on ports 80/443. A Python-managed config generator writes `server {}` blocks for each project, mapping `Host` header to the project's nginx container. The config is regenerated and nginx is reloaded whenever a project is started, stopped, created, or deleted.

**Architecture:**
```
Browser → https://local.tvmatchen.nu (port 443) → shared-proxy nginx
    → proxy_pass https://tvmatchen_nginx:443

Browser → https://porto.meetup.local (port 443) → shared-proxy nginx
    → proxy_pass https://meetup_nginx:443
```

The shared proxy connects to all project networks so it can reach each project's nginx by container name.

**Pros:**
- Uses nginx, which the team already knows and already exists in every project.
- Full control over config generation -- the Python codebase already generates nginx configs.
- No new dependencies beyond the existing nginx Docker image.
- Config is a simple, inspectable file.
- Supports per-domain SSL certs (reusing existing project certs).

**Cons:**
- Must regenerate config and reload nginx on every project start/stop/create/delete.
- Must manage Docker network connectivity (shared proxy must join each project's network).
- Slightly more Python code in a new `ProxyManager` class.

**Complexity:** Medium

### Option C: Traefik with Docker Labels

Run Traefik as a reverse proxy that auto-discovers backends via Docker labels on project containers.

**Pros:**
- Zero-config discovery: Traefik watches the Docker socket and routes based on labels.
- No config regeneration needed -- purely event-driven.
- Built-in Let's Encrypt support (irrelevant for local dev, but worth noting).

**Cons:**
- New technology the team does not know -- debugging Traefik label-based routing is non-trivial.
- Requires adding labels to every project's `docker-compose.yml` (changes existing file generation).
- Traefik's default behavior requires careful configuration for SSL passthrough vs. termination.
- Docker socket access (`/var/run/docker.sock`) is a security consideration, though acceptable for local dev.
- Error messages and logs are less intuitive than nginx for the team.

**Complexity:** Medium-High

### Option D: nginx-proxy (jwilder/nginx-proxy) with Docker Environment Variables

Use the `nginx-proxy` Docker image, which auto-generates nginx config by reading `VIRTUAL_HOST` and `VIRTUAL_PORT` environment variables from running containers.

**Pros:**
- Established project with good documentation.
- Auto-discovery via environment variables -- no manual config needed.
- Uses nginx under the hood, so the team can read the generated config.
- Companion project (`acme-companion`) handles SSL, though not needed here.

**Cons:**
- Requires adding `VIRTUAL_HOST` and `VIRTUAL_PORT` environment variables to each project's nginx container.
- SSL handling is opinionated: expects certs at specific paths inside the proxy container.
- Less control than a hand-managed nginx config -- edge cases (subfolder routing, custom headers) may be harder to handle.
- Another Docker image to maintain/update.

**Complexity:** Medium

## Recommendation

**Option B: Standalone Nginx Reverse Proxy with Generated Config.**

### Rationale

This option wins on every axis that matters for a local dev tool:

1. **Team familiarity.** The entire project already generates nginx configs. Adding one more nginx config -- this time for the proxy -- is a natural extension of existing patterns.

2. **Full control.** Unlike Traefik or nginx-proxy, we control every line of the generated config. This matters for edge cases like subfolder projects (`local.tvmatchen.nu/betting`), mixed domain patterns, and per-project SSL certs.

3. **Debuggability.** When something goes wrong, the developer can `docker exec` into the proxy container and read a plain nginx config file. No label DSL, no TOML, no magic.

4. **Minimal blast radius.** The proxy is a thin routing layer. Each project's nginx continues to handle SSL termination, PHP-FPM, rewrites, and caching. The proxy just forwards TCP streams.

5. **Simplicity of integration.** The Python codebase already has the data it needs (domain, container name, SSL cert paths) in `config.json`. Generating an nginx config from this data is straightforward.

### Why Not the Others

- **Option A (Do Nothing):** The port mismatch with WordPress `siteurl`/`home` is a real functional problem, not just a convenience issue. Redirect loops and broken assets make this untenable.
- **Option C (Traefik):** Introducing a new piece of infrastructure the team does not understand is the wrong tradeoff for a local dev tool. Debugging Traefik label routing requires specialized knowledge.
- **Option D (nginx-proxy):** Close second, but the opinionated SSL handling and reduced control over the generated config make it less suitable for this project's diverse domain patterns and subfolder routing.

## Detailed Design

### Architecture Overview

```
                        ┌─────────────────────────────┐
                        │   Shared Reverse Proxy       │
                        │   (nginx on ports 80/443)    │
                        │                              │
   Browser ──────────►  │  Host: local.tvmatchen.nu    │──► tvmatchen_nginx:443
   port 443             │  Host: porto.meetup.local    │──► meetup_nginx:443
                        │  Host: local.casimaru.test   │──► casimaru_nginx:443
                        │                              │
                        │  Default: 502 Bad Gateway    │
                        └─────────────────────────────┘
                            │         │         │
                ┌───────────┘    ┌────┘    ┌────┘
                ▼                ▼         ▼
         tvmatchen_          meetup_    casimaru_
         wordpress_network   wordpress_ wordpress_
                             network    network
```

### SSL Strategy: Terminate at the Shared Proxy, Plain HTTP to Backend

The cleanest approach is to have the shared proxy terminate SSL using each project's existing cert/key pair, then forward plain HTTP to the backend project's nginx on port 80. This avoids the complexity of SSL passthrough and double encryption.

However, this means the project-level nginx receives plain HTTP, not HTTPS. Since the project nginx already listens on both port 80 and 443, this works without modification -- the proxy simply targets port 80.

**Why not SSL passthrough?** SSL passthrough (using nginx `stream` module with `ssl_preread`) routes based on SNI. This works, but:
- It requires the `ngx_stream_ssl_preread_module`, which needs a custom nginx build or specific image.
- It routes at the TCP level, losing the ability to add headers (like `X-Forwarded-For`).
- It adds no benefit for local dev since the project certs are self-signed anyway.

**Why not proxy_pass to HTTPS backend?** This would mean `proxy_pass https://tvmatchen_nginx:443`, which works but requires the proxy to trust the backend's self-signed cert (`proxy_ssl_verify off`). This adds configuration complexity for no benefit. Plain HTTP between two containers on the same Docker network is fine for local dev.

### Docker Setup

#### File: `reverse-proxy/docker-compose.yml`

Lives at the repository root under `reverse-proxy/`.

```yaml
services:
  proxy:
    image: nginx:alpine
    container_name: wp_local_proxy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./conf.d:/etc/nginx/conf.d:ro
      - ../wordpress-projects:/etc/nginx/project-ssl:ro
    # Networks are attached dynamically -- see "Network Connectivity" below
```

The `volumes` mount maps the entire `wordpress-projects/` tree as read-only under `/etc/nginx/project-ssl/`, giving the proxy access to each project's `ssl/cert.pem` and `ssl/key.pem` at predictable paths like `/etc/nginx/project-ssl/tvmatchen/ssl/cert.pem`.

#### File: `reverse-proxy/nginx.conf`

The main nginx config (static, never regenerated):

```nginx
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 256;
}

http {
    log_format main '$remote_addr - $host [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer"';
    access_log /var/log/nginx/access.log main;

    # Reasonable defaults for local dev
    sendfile on;
    keepalive_timeout 65;
    client_max_body_size 100M;

    # Proxy settings
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_connect_timeout 10s;
    proxy_read_timeout 300s;

    # Default server: returns 502 for unknown hosts
    server {
        listen 80 default_server;
        listen 443 ssl default_server;

        ssl_certificate /etc/nginx/default-ssl/cert.pem;
        ssl_certificate_key /etc/nginx/default-ssl/key.pem;

        server_name _;

        location / {
            return 502 '{"error": "No project is running for this domain. Check that the project is started."}';
            add_header Content-Type application/json;
        }

        # Health check for the proxy itself
        location /proxy-health {
            return 200 'ok';
            add_header Content-Type text/plain;
        }
    }

    # Project-specific server blocks are loaded from conf.d/
    include /etc/nginx/conf.d/*.conf;
}
```

#### File: `reverse-proxy/conf.d/tvmatchen.conf` (generated example)

One file per project, generated by the Python `ProxyManager`:

```nginx
# Auto-generated by WordPress Local Dev -- do not edit manually
# Project: tvmatchen
# Domain: local.tvmatchen.nu
# Generated: 2026-03-27T12:00:00

# HTTP → HTTPS redirect
server {
    listen 80;
    server_name local.tvmatchen.nu;
    return 301 https://$host$request_uri;
}

# HTTPS termination and proxy to project nginx
server {
    listen 443 ssl;
    http2 on;
    server_name local.tvmatchen.nu;

    ssl_certificate /etc/nginx/project-ssl/tvmatchen/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/project-ssl/tvmatchen/ssl/key.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    location / {
        proxy_pass http://tvmatchen_nginx:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

For projects without SSL enabled, the HTTPS block is omitted and only the HTTP server block is generated.

### Network Connectivity

Each project runs in its own Docker bridge network (e.g., `tvmatchen_wordpress_network`). The shared proxy must be able to reach each project's nginx container by name. There are two strategies:

#### Strategy: Dynamic Network Attachment via Docker CLI

When a project starts, the Python code runs:
```
docker network connect tvmatchen_wordpress_network wp_local_proxy
```

When a project stops, the Python code runs:
```
docker network disconnect tvmatchen_wordpress_network wp_local_proxy
```

This is the simplest approach and requires no changes to project-level `docker-compose.yml` files.

**Why not a shared network?** A single `wp_local_proxy_network` that all projects join would also work, but it would mean modifying every project's `docker-compose.yml` to add an `external` network. The dynamic attachment approach avoids touching project files.

### Python Integration: New `ProxyManager` Class

#### File: `utils/proxy_manager.py`

```python
"""Manages the shared reverse proxy for routing by Host header."""

import json
import subprocess
import datetime
from pathlib import Path

from .docker_compose_detect import compose_command


class ProxyManager:
    """Manages the shared nginx reverse proxy that routes port 80/443
    traffic to the correct project based on the Host header."""

    PROXY_DIR_NAME = "reverse-proxy"
    PROXY_CONTAINER = "wp_local_proxy"

    def __init__(self, base_dir, projects_dir):
        self.base_dir = Path(base_dir)
        self.projects_dir = Path(projects_dir)
        self.proxy_dir = self.base_dir / self.PROXY_DIR_NAME

    def ensure_proxy_running(self):
        """Start the shared proxy if it is not already running."""
        if not self.proxy_dir.exists():
            self._create_proxy_infrastructure()

        # Check if container is running
        result = subprocess.run(
            ["docker", "inspect", "-f", "{{.State.Running}}",
             self.PROXY_CONTAINER],
            capture_output=True, text=True, timeout=10
        )
        if result.returncode != 0 or result.stdout.strip() != "true":
            subprocess.run(
                compose_command("up", "-d"),
                cwd=self.proxy_dir,
                capture_output=True, text=True, timeout=60
            )
        return {"success": True}

    def regenerate_config(self):
        """Scan all projects and regenerate proxy nginx configs."""
        conf_dir = self.proxy_dir / "conf.d"
        conf_dir.mkdir(parents=True, exist_ok=True)

        # Remove old generated configs
        for f in conf_dir.glob("*.conf"):
            f.unlink()

        # Scan projects
        for project_dir in sorted(self.projects_dir.iterdir()):
            if not project_dir.is_dir():
                continue
            config_file = project_dir / "config.json"
            if not config_file.exists():
                continue

            config = json.loads(config_file.read_text())
            self._write_project_config(conf_dir, config, project_dir.name)

        # Reload nginx (graceful, no downtime)
        self._reload_nginx()

    def on_project_start(self, project_name, project_config):
        """Called after a project's containers start."""
        self.ensure_proxy_running()

        # Connect proxy to project network
        network_name = f"{project_name}_wordpress_network"
        subprocess.run(
            ["docker", "network", "connect", network_name,
             self.PROXY_CONTAINER],
            capture_output=True, text=True, timeout=15
        )
        # Errors are non-fatal (may already be connected)

        # Regenerate and reload
        self.regenerate_config()

    def on_project_stop(self, project_name):
        """Called after a project's containers stop."""
        # Disconnect proxy from project network (network may already
        # be removed by docker-compose down, so errors are expected)
        network_name = f"{project_name}_wordpress_network"
        subprocess.run(
            ["docker", "network", "disconnect", network_name,
             self.PROXY_CONTAINER],
            capture_output=True, text=True, timeout=15
        )

        # Remove project config and reload
        conf_file = self.proxy_dir / "conf.d" / f"{project_name}.conf"
        if conf_file.exists():
            conf_file.unlink()
        self._reload_nginx()

    def on_project_delete(self, project_name):
        """Called when a project is fully deleted."""
        self.on_project_stop(project_name)

    def _write_project_config(self, conf_dir, config, project_name):
        """Write an nginx server block config for one project."""
        domain = config.get("domain", "").split("/")[0]
        enable_ssl = config.get("enable_ssl", True)
        nginx_container = f"{project_name}_nginx"
        now = datetime.datetime.now().isoformat()

        if not domain:
            return

        lines = [
            f"# Auto-generated by WordPress Local Dev -- do not edit",
            f"# Project: {project_name}",
            f"# Domain: {domain}",
            f"# Generated: {now}",
            "",
        ]

        if enable_ssl:
            # HTTP to HTTPS redirect
            lines += [
                "server {",
                "    listen 80;",
                f"    server_name {domain};",
                "    return 301 https://$host$request_uri;",
                "}",
                "",
                "server {",
                "    listen 443 ssl;",
                "    http2 on;",
                f"    server_name {domain};",
                "",
                f"    ssl_certificate /etc/nginx/project-ssl/{project_name}/ssl/cert.pem;",
                f"    ssl_certificate_key /etc/nginx/project-ssl/{project_name}/ssl/key.pem;",
                "    ssl_protocols TLSv1.2 TLSv1.3;",
                "    ssl_ciphers HIGH:!aNULL:!MD5;",
                "",
                "    client_max_body_size 100M;",
                "",
                "    location / {",
                f"        proxy_pass http://{nginx_container}:80;",
                "        proxy_set_header Host $host;",
                "        proxy_set_header X-Real-IP $remote_addr;",
                "        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;",
                "        proxy_set_header X-Forwarded-Proto https;",
                "    }",
                "}",
            ]
        else:
            lines += [
                "server {",
                "    listen 80;",
                f"    server_name {domain};",
                "",
                "    client_max_body_size 100M;",
                "",
                "    location / {",
                f"        proxy_pass http://{nginx_container}:80;",
                "        proxy_set_header Host $host;",
                "        proxy_set_header X-Real-IP $remote_addr;",
                "        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;",
                "        proxy_set_header X-Forwarded-Proto http;",
                "    }",
                "}",
            ]

        (conf_dir / f"{project_name}.conf").write_text("\n".join(lines) + "\n")

    def _reload_nginx(self):
        """Send SIGHUP to nginx inside the proxy container for zero-
        downtime config reload."""
        subprocess.run(
            ["docker", "exec", self.PROXY_CONTAINER, "nginx", "-s", "reload"],
            capture_output=True, text=True, timeout=10
        )

    def _create_proxy_infrastructure(self):
        """Create the proxy directory, docker-compose.yml, and
        default nginx.conf if they do not exist."""
        self.proxy_dir.mkdir(parents=True, exist_ok=True)
        (self.proxy_dir / "conf.d").mkdir(exist_ok=True)

        # Default SSL cert for the catch-all server block
        default_ssl_dir = self.proxy_dir / "default-ssl"
        default_ssl_dir.mkdir(exist_ok=True)
        # Generate a minimal self-signed cert (or copy from first project)
        self._generate_default_ssl(default_ssl_dir)

        # docker-compose.yml
        compose = self._get_docker_compose_content()
        (self.proxy_dir / "docker-compose.yml").write_text(compose)

        # nginx.conf
        nginx_conf = self._get_nginx_conf_content()
        (self.proxy_dir / "nginx.conf").write_text(nginx_conf)

    # --- Templates (abbreviated here, see full versions in design) ---

    def _get_docker_compose_content(self):
        return """services:
  proxy:
    image: nginx:alpine
    container_name: wp_local_proxy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
      - ./conf.d:/etc/nginx/conf.d:ro
      - ./default-ssl:/etc/nginx/default-ssl:ro
      - ../wordpress-projects:/etc/nginx/project-ssl:ro
"""

    def _get_nginx_conf_content(self):
        return """worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 256;
}

http {
    log_format main '$remote_addr - $host [$time_local] "$request" '
                    '$status $body_bytes_sent';
    access_log /var/log/nginx/access.log main;

    sendfile on;
    keepalive_timeout 65;
    client_max_body_size 100M;

    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_connect_timeout 10s;
    proxy_read_timeout 300s;

    server {
        listen 80 default_server;
        listen 443 ssl default_server;
        ssl_certificate /etc/nginx/default-ssl/cert.pem;
        ssl_certificate_key /etc/nginx/default-ssl/key.pem;
        server_name _;

        location / {
            return 502 'No project is running for this domain.';
            add_header Content-Type text/plain;
        }
        location /proxy-health {
            return 200 'ok';
            add_header Content-Type text/plain;
        }
    }

    include /etc/nginx/conf.d/*.conf;
}
"""

    def _generate_default_ssl(self, ssl_dir):
        """Generate a throwaway self-signed cert for the default
        server block. Only used for unknown-host 502 responses."""
        cert_file = ssl_dir / "cert.pem"
        key_file = ssl_dir / "key.pem"
        if cert_file.exists() and key_file.exists():
            return

        from cryptography import x509
        from cryptography.x509.oid import NameOID
        from cryptography.hazmat.primitives import hashes, serialization
        from cryptography.hazmat.primitives.asymmetric import rsa
        import datetime as dt

        key = rsa.generate_private_key(public_exponent=65537, key_size=2048)
        subject = issuer = x509.Name([
            x509.NameAttribute(NameOID.COMMON_NAME, "wp-local-proxy-default"),
        ])
        cert = (
            x509.CertificateBuilder()
            .subject_name(subject)
            .issuer_name(issuer)
            .public_key(key.public_key())
            .serial_number(x509.random_serial_number())
            .not_valid_before(dt.datetime.now(dt.timezone.utc))
            .not_valid_after(
                dt.datetime.now(dt.timezone.utc) + dt.timedelta(days=3650)
            )
            .sign(key, hashes.SHA256())
        )
        key_file.write_bytes(
            key.private_bytes(
                serialization.Encoding.PEM,
                serialization.PrivateFormat.PKCS8,
                serialization.NoEncryption(),
            )
        )
        cert_file.write_bytes(cert.public_bytes(serialization.Encoding.PEM))
```

#### `project_manager.py` -- `stop_project()`

```python
def stop_project(self, project_name):
    project_path = self.projects_dir / project_name

    # NEW: update reverse proxy BEFORE stopping (while network exists)
    # Actually, docker-compose down removes the network, so we
    # regenerate config AFTER stopping and let disconnect fail gracefully.
    result = self.docker_manager.stop_project(project_path)

    if result.get('success'):
        self.proxy_manager.on_project_stop(project_name)

    return result
```

### Handling Existing High-Numbered Ports

The high-numbered port mappings (HTTP_PORT, HTTPS_PORT) remain in each project's Docker Compose and `.env`. They serve as a **fallback direct-access path** -- useful for debugging the proxy itself or accessing a project when the proxy is down.

The shared proxy does NOT use these port numbers. It routes to `{name}_nginx:80` via Docker networking. The host port mappings are irrelevant to the proxy.

This means both access paths work simultaneously:
- `https://local.tvmatchen.nu` (port 443, via shared proxy) -- **primary path**
- `https://local.tvmatchen.nu:10091` (direct to project nginx) -- **fallback/debug path**

### Handling Subfolder Projects

Projects like tvmatchen with `domain: local.tvmatchen.nu/betting` have a subfolder component. The shared proxy routes by `Host` header only (which is `local.tvmatchen.nu`, without the path). The `/betting` path routing is already handled by the project's own nginx config. The proxy simply forwards everything under that host:

```nginx
server_name local.tvmatchen.nu;
location / {
    proxy_pass http://tvmatchen_nginx:80;
    ...
}
```

The project nginx then handles `/betting/` rewriting to WordPress. No special subfolder handling is needed at the proxy level.

### Proxy Lifecycle and Resilience

1. **First project start**: `ensure_proxy_running()` creates the infrastructure and starts the proxy container.
2. **Subsequent starts**: `ensure_proxy_running()` sees the container is running and does nothing.
3. **All projects stopped**: The proxy remains running with an empty `conf.d/`. Requests get the default 502 response.
4. **System reboot**: The proxy container has `restart: unless-stopped`. Docker Desktop will restart it automatically.
5. **Proxy manually stopped**: The next `start_project()` call will restart it via `ensure_proxy_running()`.
6. **Proxy crash**: Docker's restart policy handles this. The next project operation also checks.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Port 80/443 already in use on the host (e.g., Apache, another nginx, macOS web sharing) | Medium | High | Check on startup and provide a clear error message with instructions to free the ports. Fallback: configurable proxy ports (e.g., 8080/8443). |
| Docker network connect fails (race condition between project start and network creation) | Low | Medium | Retry network connect with a short delay. The `docker-compose up -d` call creates the network before containers start, so this is unlikely. |
| Proxy nginx config has syntax error from malformed project data | Low | Medium | Run `nginx -t` before reload. If validation fails, skip the bad config and log a warning. |
| Proxy container crashes and is not restarted before a request arrives | Low | Low | Docker `restart: unless-stopped` handles this. Additionally, `ensure_proxy_running()` is called on every project operation. |
| macOS port 80/443 require elevated privileges | Low | High | Docker Desktop for Mac maps host ports through its Linux VM, not directly on macOS. No sudo needed. Verify this works and document any exceptions. |
| SSL cert paths break if project directory is renamed or moved | Very Low | Medium | Proxy config uses project name from `config.json`, which matches the directory name by convention. Document that renaming project directories requires regenerating proxy config. |
| `docker-compose down` removes the project network before proxy can disconnect | Medium | None | `network disconnect` failing is harmless -- the network is already gone. The `on_project_stop()` method handles this gracefully. |

## Alternatives Considered but Rejected

### DNS-Based Routing (dnsmasq / CoreDNS)

Using a local DNS server to resolve `*.test` or `*.local` domains to 127.0.0.1 was considered as a complement to the hosts file. While this would eliminate manual hosts file editing, it does not solve the port routing problem -- DNS can only resolve to an IP, not an IP:port. A reverse proxy is still needed regardless of how DNS is configured.

This could be a future enhancement (ADR-004) that replaces the `HostsManager` with a local DNS container, but it is orthogonal to the proxy decision.

### Modifying Project Nginx to Listen on Port 80/443 via a Shared External Network

Instead of a separate proxy, each project's nginx could join a shared external network and listen on port 80/443, using nginx's `server_name` to differentiate. This was rejected because:
- Only one container can bind a given host port -- having multiple nginx containers all trying to bind port 443 does not work.
- Even with a shared network, Docker port mapping is per-container, not per-network.

### Using `host` Network Mode

Running project nginx containers with `network_mode: host` would let them all share the host network stack, but only one could bind port 443. This does not solve the multi-project problem.

## Consequences

**Positive:**
- Developers access sites at clean URLs: `https://local.tvmatchen.nu`, `https://porto.meetup.local`.
- WordPress `siteurl` and `home` options work correctly without port numbers.
- Multiple projects run simultaneously on standard ports -- the original promise of ADR-001 is fully realized.
- No changes to existing per-project Docker Compose files or nginx configs.
- The proxy is fully automated -- no manual configuration needed.

**Negative:**
- One additional long-running Docker container (`wp_local_proxy`) consumes ~5MB memory.
- Ports 80 and 443 on the host are now occupied by the proxy -- if the developer runs another web server locally, it will conflict. (Mitigated: the proxy only needs to run when projects are active.)
- Slight increase in request latency (one extra network hop through the proxy). Negligible for local dev.
- Config regeneration on every start/stop adds ~100ms to the operation.

**Neutral:**
- High-numbered ports continue to work as a direct-access fallback.
- The proxy infrastructure is self-contained in `reverse-proxy/` and can be removed without affecting projects.

## ADR Record

**Decision:** Deploy a standalone nginx reverse proxy container (`wp_local_proxy`) on ports 80/443 that routes requests by `Host` header to the correct project's nginx container. The proxy terminates SSL using each project's existing certificates and forwards plain HTTP to the backend. A new `ProxyManager` Python class regenerates the proxy nginx config and reloads nginx whenever a project is started, stopped, created, or deleted. Docker network connectivity is managed dynamically via `docker network connect/disconnect`.

**Status:** Proposed

**Consequences:** Developers can access all running WordPress projects at standard URLs (e.g., `https://local.tvmatchen.nu`) without port numbers. The proxy adds one lightweight container and occupies host ports 80/443. Existing project configurations are unchanged. High-numbered direct-access ports remain as a fallback.
